# C.U.T.E Rubric (0-100)

Score = C + U + T + E (max 100)
- C (Completeness) 0..25
- U (Unambiguity)  0..25
- T (Testability)  0..25
- E (EARS)         0..25

## C: Completeness (25)
減点例:
- 必須セクション欠落（目的/スコープ/用語集/前提/機能要件/非機能要件/セキュリティ/運用/未解決事項）: -3/セクション
- 各REQに必須フィールド欠落（種別/優先度/要件文/受入テスト/例外）: -1/欠落
- 未解決事項(Open Questions)が1件以上: -2/件（=100点不可の根拠）

## U: Unambiguity (25)
- banned_words.txt の語が出現: -1/出現（上限25）
- "適切/最適/よしなに/できるだけ/可能な限り/高速/適宜/ユーザーフレンドリー"など主観語は全て減点対象

## T: Testability (25)
- 受入テストが無いREQ: -3/REQ
- GWT(Given/When/Then)形式でない: -1/REQ
- 期待結果が客観的でない（数値/状態/YesNoで判定できない）: -1/REQ（機械的には検出しきれないため、主にレビューで補助）

## E: EARS Compliance (25)
- 要件文(EARS)がEARSパターンに一致しないREQ: -2/REQ

## 採点フロー

```
1. requirements.md を読み込み
2. 必須セクションの存在チェック → C減点
3. 各REQブロックをパース
   - 必須フィールド(種別/優先度/要件文/受入テスト/例外)チェック → C減点
   - 要件文がEARSパターンに一致するかチェック → E減点
   - 受入テストがGWT形式かチェック → T減点
4. 全文から banned_words をカウント → U減点
5. 未解決事項セクションの項目数をカウント → C減点
6. 各次元を0..25にクランプして合計
```

## 100点の意味

100点 = 以下がすべて満たされている状態:
- 必須セクションがすべて存在
- 全REQに必須フィールドが存在
- 全REQの要件文がEARSパターンに一致
- 全REQに受入テスト(GWT形式)が存在
- 曖昧語が0個
- 未解決事項が0件

## 98点以上を目標とする理由

- 人間のレビューを経ていない段階では、軽微な改善余地が残る可能性がある
- 98点以上であれば「実装可能な品質」とみなす
- 90点未満は「要件定義として不十分」とみなし、改善ループを継続
